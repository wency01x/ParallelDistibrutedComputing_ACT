# My Reflection
---
Doing this activity with Kerby and Wency really helped me understand how sequential and parallel codes actually run. At first, doing the sequential sorting was easy because it just follows a single linear control flow. When we tested it on the small dataset of 1,000 elements , the sequential method was actually really fast and had almost no overhead. But when we tried the large dataset with 1,000,000 elements, it took a very long time. That is when we saw the power of parallel execution. By dividing the problem into smaller chunks, the parallel code finished way faster on the big data. But honestly, writing the parallel version was very hard. We had to use Python multiprocessing to split the work and then figure out how to correctly merge the results back into one sorted output. I realized that setting up multiple processes creates a lot of overhead. The computer spends extra time just managing tasks and handling synchronization to avoid race conditions. Because of this extra work, using parallel code on the small dataset was slower and totally unnecessary. I learned that you only really need parallel programming when you are dealing with massive amounts of data where the computing workload is bigger than the coordination costs.
