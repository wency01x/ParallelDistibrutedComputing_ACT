Working with Harvie and Kerby on this activity really helped me understand the difference between sequential and parallel algorithms in a practical way. My part was building the dataset generator and both sequential algorithms. Writing the sequential merge sort was something I found manageable at first because it just follows a straightforward recursive pattern where you split the list, sort each half, and merge them back together. When we tested it on the small dataset of 1,000 elements it worked really fast and there was barely any overhead at all. But when we ran it on the large dataset with 1,000,000 elements it took noticeably longer and that is when I started to understand why parallel execution exists. I also implemented the sequential linear search which was even simpler since it just scans the list one element at a time until it finds the target. What I noticed is that for searching, the sequential version was actually faster than the parallel one even on large data because each comparison is so cheap that spawning multiple processes just adds unnecessary cost. The hardest part for me was making sure my dataset generator was structured cleanly so that the other members could import it into their files without any problems. I also had to make sure my merge sort was completely correct before Harvie could use it inside the parallel sort worker. Through this activity I learned that sequential algorithms are not worse than parallel ones, they are just better suited for smaller workloads or tasks where the work per element is simple. Parallelism only really makes sense when the data is large enough that the time saved by splitting the work is greater than the time lost managing all the processes.
